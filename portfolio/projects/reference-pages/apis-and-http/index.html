Web APIs & HTTP

What is HTTP?

HTTP stands for Hypertext Transfer Protocol, and is used to structure requests and responses over the internet. HTTP requires that data is transferred from one point to another over the network.

What is TCP?

TCP (Transmission Control Protocol) is what actually handles the transfer of data. TCP is used to manage many types of internet connections in which one computer wants to send data to another.

How do HTTP and TCP work together?

When you type an address into the browser to load a webpage, you command TCP to open a connection between your browser (the client) and the server to which the url relates to. Once the TCP connection is established, the client sends a HTTP GET request to request the webpage. With HTTP/1.0 (the first version of HTTP), once the browser sends back a response, the TCP connection is closed.

What are APIs?

An Application Programming Interface (API) is a software tool that makes it easier for developers to interact with another application to use some of that application's functionality.

Types of Web APIs

Browser APIs

Browser APIs are specific to writing code that interacts with the browser. See MDN's documentation of web (browser) APIs

3rd Part APIs

These are APIs provided by third parties, usually companies. Organizations that maintain third-party APIs often set their own rules and requirements for how users utilize their API.

What is REST?

REST, or REpresentational State Transfer, is an architectural style for providing standards between computer systems on the web, making it easier for systems to communicate with each other. REST-compliant systems, often called RESTful systems, are characterized by how they are stateless and separate the concerns of client and server.

Separation of Client and Server

In the REST architectural style, the implementation of the client and the implementation of the server can be done independently without each knowing about the other. This means that the code on the client side can be changed at any time without affecting the operation of the server, and the code on the server side can be changed without affecting the operation of the client. As long as each side knows what format of messages to send to the other, they can be kept modular and separate. Separating the user interface concerns from the data storage concerns, we improve the flexibility of the interface across platforms and improve scalability by simplifying the server components. Additionally, the separation allows each component the ability to evolve independently.

Statelessness

Systems that follow the REST paradigm are stateless, meaning that the server does not need to know anything about what state the client is in and vice versa. In this way, both the server and the client can understand any message received, even without seeing previous messages. This constraint of statelessness is enforced through the use of resources, rather than commands. Resources are the nouns of the Web - they describe any object, document, or thing that you may need to store or send to other services.

Communication between Client and Server

In REST architecture, clients send requests to retrieve or modify resources, and serverse respond to those requests.

Making Requests

REST requires that the client makes requests to retrieve or modify a resource. This request will include:

  • An HTTP verb, which defines the operation to perform.
  • A header, which allows the client to pass along information about the request
  • A path to the resource
  • An optional body containing data

HTTP Verbs

There are 4 basic HTTP verbs in a REST system:

  • GET - retrieve a specific resource by id (or a collection of resources)
  • POST - create a new resource
  • PUT - update a resource
  • DELETE - delete a resource

Headers and Accept parameters

In the header of the request, the client specifies the type of content that it can receive. This is called the Accept field. The values provided in the accept field are MIME types (or Multipurpose Internet Mail Extensions). MIME types consist of a type and a subtype. text/html and image/png are examples of MIME types.

Below is an example request for a resource with id 5 in a cars resource on a server:

GET /cars/5
Accept: text/json text/html

Paths

Paths are required in a request, and they indicate where to find the requested resource. In the example request above, the path is /cars/5

Sending Responses

When sending a response back to the client, the server must include a content-type in the header of the response. These content-types are also MIME Types, and they tell the client what type of content to expect. The server must also include a response code to let the user know of the outcome.

Status Code: 200 (OK)
Content-type: application/json
